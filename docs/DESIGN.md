# 대한민국 2026 월드컵 32강 진출 트래커 — 설계 문서

> 작성 기준: 2026년 6월 26일 대화 합의 사항  
> 상태: 구현 전 설계 확정

---

## 1. 프로젝트 개요

### 1.1 목적

2026 FIFA 월드컹 조별리그 막바지, **대한민국(A조 3위)이 「조 3위 중 상위 8팀」에 들어 32강에 진출할 수 있는지**를 추적하고, 관련 경기 결과에 따라 구독자에게 알림을 제공한다.

### 1.2 배경

- 대한민국은 **A조 3위**(승점 3, 득실 -1)로 조별리그를 마침.
- 12개 조 중 각 조 1·2위 + **조 3위 중 상위 8팀**이 32강 진출.
- 다른 조의 최종 경기 결과에 따라 대한민국의 32강 가능 여부가 결정됨.

### 1.3 핵심 일정 (KST)

| # | 경기 | 일시 (KST) | 필요 결과 |
|---|------|------------|-----------|
| 1 | 세네갈 vs 이라크 (I조) | 6/27(토) 04:00 | 세네갈 **1골차 이하** 승리 **또는** 이라크 **4골차 이하** 승리 *(달성 확률 높음)* |
| 2 | 우루과이 vs 스페인 (H조) | 6/27(토) 09:00 | **스페인 승리** (스페인 조 1위 확정 동기) |
| 3 | 이집트 vs 이란 (G조) | 6/27(토) 12:00 | **이집트 승리** |
| 4 | 크로아티아 vs 가나 (L조) | 6/28(일) 06:00 | **가나 승리** |
| 5 | 콩고민주공화국 vs 우즈베키스탄 (K조) | 6/28(일) 08:30 | 콩고민주공화국 **무승부 또는 패배** |
| 6 | 알제리 vs 오스트리아 (J조) | 6/28(일) 11:00 | **오스트리아 승리** 또는 알제리 **2골차 이상** 대승 |

|------------------|----------------------|

## 2. 채택하지 않은 방안과 사유

### 2.1 카카오 i 오픈빌더 챗봇

| 항목 | 내용 |
|------|------|
| 장점 | Event API로 카카오톡 푸시 가능, 채널 친구 대상 능동 발송 |
| 병목 | OBT 승인 **영업일 3~7일** (평균 약 5일), 반려 시 재심사 추가 |
| 결론 | 6/26 신청 시 6/28 결정적 경기 전 승인 어려움 → **채택 안 함** |

### 2.2 알림톡

- 사업자등록·비즈니스 채널·템플릿 심사·딜러사 계약 필요.
- 개인 취미/단기 프로젝트에 부적합 → **채택 안 함**.

### 2.3 일반 카카오톡 단톡방 알림

- 공식 API 없음. 비공식 봇(메신저봇R 등)은 약관 위반·계정 정지 위험 → **채택 안 함**.

### 2.4 Canvas만 단독 사용

- 실시간 연동·푸시 불가, 수동 갱신 위주.
- 보조 UI로는 가능하나 **알림 수단으로는 부족** → 단독 채택 안 함.

## 3. 채택 방안: 하이브리드 (웹 + 카카오 메모 + Web Push)

### 3.1 한 줄 요약

```
카카오 로그인 구독자 전원 → 나와의 채팅에 상세 메시지 (나에게 보내기 API)
Web Push 가능 구독자     → 짧은 제목 푸시 (벨 역할)
푸시 탭                  → 브릿지 페이지 → 카카오톡 열기 유도
Web Push 불가 구독자     → 메모 + 웹 대시보드만
```

### 3.2 역할 분담

| 수단 | 역할 | 카카오톡 앱 푸시 |
|------|------|------------------|
| **나에게 보내기 API** | 상세 현황을 「나와의 채팅」에 기록 | ❌ (카카오 정책: 메모용, 알림 없음) |
| **Web Push** | 「지금 확인하라」는 짧은 제목 알림 | ✅ (폰 알림센터, 카톡 앱 밖) |
| **웹 대시보드** | 전체 현황·가이드·폴백 | — |
| **브릿지 페이지 `/go`** | 푸시 탭 후 카톡/웹 안내 | — |

> **카카오 데브톡 공식 답변:** 「나에게 보내기」는 사용자가 이미 인지한 내용을 나와의 채팅에 메모하는 스펙이며, **알림은 발생하지 않는다.**

## 4. 시스템 아키텍처

### 4.1 구성 요소

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  웹 클라이언트 │────▶│  API 서버     │────▶│  KickoffAPI     │
│  (구독/대시보드)│     │  (판정·발송)  │     │  (§8.3 종료 후  │
│               │     │              │     │   폴링)         │
└─────────────┘     └──────┬───────┘     └─────────────────┘
        │                  │
        │                  ├──▶ 카카오 memo API (구독자별)
        │                  └──▶ Web Push (endpoint별)
        │
        └── Service Worker (Web Push 수신)
```

### 4.2 구독 플로우

1. 사용자가 웹 접속 (환경 감지 → 가이드 표시)
2. **카카오 로그인** + `talk_message` 동의 (`scope=talk_message` 명시)
3. 서버에 **refresh token** 저장
4. (환경 허용 시) **Web Push 구독** → endpoint를 서버에 저장
5. 구독 완료

### 4.3 알림 발송 플로우

```
[서버] API 폴링으로 경기 종료(FT) 감지 (§8.3)
   │
   ├─① 최종 스코어 확보 → MatchState 갱신 → 6가지 조건 판정
   │
   ├─② [카카오 구독자 전원] — 경기 종료 직후·조건 수 변화 시
   │     POST /v2/api/talk/memo/default/send (또는 사용자 정의 템플릿)
   │     → 「나와의 채팅」에 **종료 스코어** + 6조건 현황 피드 카드
   │
   └─③ [Web Push 구독자]
         제목: 예) "세네갈 1-0 이라크 종료 · 32강 2/3"
         본문: 비우거나 한 줄 (충족 수 요약)
         data.url: /go?status=...
         → notificationclick 시 브릿지 페이지 오픈
```

### 4.4 브릿지 페이지 (`/go`)

푸시 탭 시 **나와의 채팅으로 직접 딥링크는 불가** (공식 스킴 없음). 대신 중간 페이지 제공.

**표시 내용 예:**
- 현재 충족 조건 수 (예: 2/3)
- 「상세 내용은 카카오톡 나와의 채팅에 보내 두었습니다」
- **[카카오톡 열기]** → `kakaotalk://launch` (사용자 탭으로 실행)
- **[웹에서 보기]** → 대시보드 (폴백)

> `kakaotalk://launch`는 앱 초기 화면만 열림. 나와의 채팅 직행은 사용자가 1탭 추가.

### 4.5 나와의 채팅 메시지 템플릿 (예시)

**경기 종료 직후 (해당 경기 스코어 포함):**

```
제목: I조 종료 — 세네갈 1-0 이라크

설명:
  📊 최종 스코어: 세네갈 1 - 0 이라크
  ✅ 이 경기 조건: 충족
  📈 32강 진출 현황: 2/3 달성

  ✅ I조: 세네갈 vs 이라크 (충족)
  ✅ H조: 스페인 승리 (충족)
  ⏳ G조: 이집트 vs 이란 (대기)
  ...

[웹에서 전체 보기] → https://(서비스 URL)
```

## 5. 카카오 연동 상세

### 5.1 사용 API

| API | 용도 |
|-----|------|
| 카카오 로그인 | 구독·토큰 발급 |
| 나에게 기본/사용자 정의 템플릿 메시지 발송 | `POST /v2/api/talk/memo/default/send` |
| (선택) 나에게 스크랩 메시지 | 링크 미리보기 카드 |

### 5.2 필수 설정

- [Kakao Developers](https://developers.kakao.com/) 앱 생성
- Redirect URI 등록
- 동의 항목: **카카오톡 메시지 전송 (`talk_message`)** — 로그인 시 `scope=talk_message` 포함
- 메시지 템플릿 구성 (피드 카드 등)
- 서버에서 refresh token으로 access token 갱신

### 5.3 제약

- 사용자마다 **개별 로그인·동의** 필요
- **나와의 채팅**으로만 발송 (단톡·채널 아님)
- **카카오톡 앱 푸시 없음** → Web Push가 벨 역할
- 토큰 만료·철회 시 재로그인 필요
- 월간 무료 쿼터 내 사용 (나에게 보내기는 일일 친구 발송 쿼터와 별도)

### 6.2 구현 요건

- HTTPS 필수
- Service Worker + VAPID 키
- `web-push` 등으로 서버 발송

### 6.3 푸시 페이로드 원칙

- **제목**: 경기 종료 스코어 + 충족 수 (예: `세네갈 1-0 이라크 종료 · 32강 2/3`, `32강 확정!`)

- **클릭 URL**: `/go?...` 브릿지 페이지

## 7. 사용자 환경별 가이드 (웹 UI 제공)

구독 전 **환경 감지** 후 해당 가이드를 표시한다.

### A. Android + Chrome (권장)

1. Chrome에서 사이트 직접 접속 (카톡·네이버 인앱 아님)
2. 카카오 로그인 → 메시지 전송 동의
3. 「알림 받기」→ 허용
4. 완료

### B. iPhone (홈 화면 필수)

1. Safari에서 접속
2. 카카오 로그인 → 메시지 전송 동의
3. **공유 → 홈 화면에 추가**
4. **홈 화면 아이콘**으로 실행 (브라우저 탭 아님)
5. 「알림 받기」→ 허용
6. 완료

### C. 카카오톡·네이버 인앱으로 링크 연 경우

```
⚠️ 이 브라우저에서는 알림을 받을 수 없습니다.

[Chrome / Safari에서 열기]
```

- Android: `kakaotalk://web/openExternal?url=` + encodeURIComponent(현재 URL)
- 또는 intent 스킴으로 외부 브라우저 유도

### D. Web Push 불가, 카카오만 연동

- 나와의 채팅에만 메시지 적재 (**푸시 없음**)
- 안내: 「카카오톡 나와의 채팅을 확인하거나, 웹을 북마크해 주세요」
- 웹 대시보드에서 언제든 현황 확인 가능

## 8. 핵심 비즈니스 로직

### 8.1 조건 판정

6개 경기 결과(스코어)를 입력받아 각 조건의 충족 여부를 boolean으로 계산.

**주의:** 단순 승패가 아닌 **골차 조건**이 있는 항목 (1번, 6번)은 스코어 기반 분기 필요.

```text
조건1: (세네갈 승 && 골차 <= 1) || (이라크 승 && 골차 <= 4)
조건2: 스페인 승
조건3: 이집트 승
조건4: 가나 승
조건5: 콩고 무승부 || 콩고 패
조건6: 오스트리아 승 || (알제리 승 && 알제리 골차 >= 2)

충족 수 >= 3 → 32강 진출 가능 (최종 확정은 FIFA 조 3위 중 상위 8팀 규칙 전체 반영 필요)
```

### 8.2 발송 트리거

- 관련 경기 **종료** 시 — API에서 `FT` / `AET` / `PEN` 확인 **즉시** (§8.3)
  - 알림 본문에 **해당 경기 최종 스코어**(예: `세네갈 1-0 이라크`)를 반드시 포함
  - 이어서 6조건 판정 결과·충족 수(예: `2/3`)를 함께 전달
- 충족 조건 **개수 변화** 시 (경기 종료 알림과 동시에 발생할 수 있음 → 한 번에 발송)
- (선택) 「32강 확정」「탈락 확정」 등 이정표 도달 시

중복 발송 방지: 마지막 발송 상태(충족 수, 경기별 결과)를 저장하고 **변화 있을 때만** 발송.

### 8.3 경기 데이터 API 및 폴링

#### 8.3.1 채택 API

| 항목 | 내용 |
|------|------|
| 제공자 | **KickoffAPI** |
| Base URL | `https://api.kickoffapi.com/api/v1` |
| 인증 | 헤더 `x-api-key` |
| 대회 | `league=1`, `season=2026` (FIFA World Cup) |
| 사용 엔드포인트 | `GET /fixtures?id={api_fixture_id}` |

응답은 플랫 JSON (`id`, `statusShort`, `goalsHome`/`goalsAway`, `homeTeam`/`awayTeam`). 날짜 조회 시 KickoffAPI는 UTC 기준이므로 KST·UTC 양쪽 날짜를 조회한다 (`kickoffToApiQueryDates`).

한 번의 요청 응답에 **최종 스코어**와 **경기 상태**(`statusShort`)가 함께 포함된다.

#### 8.3.2 설계 원칙

- **킥오프·종료 예상 시각**은 §1.3 일정(KST)을 사용한다. 킥오프 직후부터의 라이브 폴링은 하지 않는다.
- **킥오프 + 110분**부터 API로 종료·스코어를 확인한다 (§8.3.3).
- 경기당 fixture ID는 서버 기동 시(또는 배포 시) `fixtures?league=1&season=2026` + 날짜로 조회해 config에 고정한다.

#### 8.3.3 폴링 스케줄 (경기당)

| 항목 | 값 |
|------|-----|
| 경기 최소 소요 가정 | 킥오프 후 **110분** 경과 시점부터 조회 시작 (하프타임 포함 벽시계 기준) |
| 첫 API 조회 | `kickoff_kst + 110분` |
| 조회 간격 | **1분** |
| 최대 조회 횟수 | **30회** (`kickoff + 140분`까지) |
| 중단 조건 | `statusShort` ∈ `{ FT, AET, PEN }` 이고 골이 null이 아닐 때 **즉시 중단** |
| 진행 중 | `1H`, `2H`, `HT`, `ET` 등 → 다음 1분 후 재조회 |
| 상한 초과 | 30회 후에도 미종료 → 수동 입력 폴백·운영 알림 (§11) |

```text
kickoff = §1.3 KST
poll_at = kickoff + 110min

repeat 최대 30회 (또는 FT 확인 시 break):
  GET /api/v1/fixtures?id={api_fixture_id}
  if FT/AET/PEN and goals 확정:
    → MatchState 갱신 → §8.1 판정 → §8.2 알림 (스코어 포함)
    break
  poll_at += 1min
```

#### 8.3.4 일일 한도 적합성

KickoffAPI 플랜 한도 내 운영 (검증 기준 일 **~100,000 req**).

| 시나리오 | 일일 요청 (경기 3개/일) |
|----------|-------------------------|
| 최악 (경기당 30회 × 3 + 기동 6) | ~96회 |
| 현실적 (`FT` 확인 후 중단) | ~10~15회 |

6경기·2일 운영 시 한도 대비 여유가 충분하다.

#### 8.3.5 폴백

- API 장애·한도 초과·30회 상한 초과: 대시보드에 **「조회 실패」** 표시
- 관리자는 `/admin` 페이지 또는 `POST /api/admin/score`로 **수동 스코어 입력** → `MatchState` 갱신 후 동일 알림 플로우 실행 (Phase 3~)

## 12. 기대 효과와 한계

### 기대

- OBT 없이 **당일~1일 내** 프로토타입 가능
- 카카오 구독자: **나와의 채팅**에 상세 기록
- Web Push 가능자: **실시간에 가까운 폰 알림**

### 한계

- 나와의 채팅만으로는 **카톡 푸시 없음**
- 푸시 탭 → **나와의 채팅 직행 불가**
- iOS·인앱 브라우저 사용자는 **가이드 없으면 Push 불가**
- 「조 3위 중 상위 8팀」 최종 순위는 FIFA 규칙 전체 시뮬레이션 필요 (다른 3위 팀과의 비교)
